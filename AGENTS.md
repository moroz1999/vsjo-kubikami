# Codex Rules

1. Никогда не удалять пользовательские переносы строк.
2. Никогда не удалять пользовательские комментарии.
3. Не менять логику при запросе на форматирование или комментирование: в таких задачах править только оформление, пустые строки и комментарии.
4. В asm-файлах комментарии по умолчанию писать к смысловым блокам, а не к каждой строке подряд. Комментарии должны объяснять ход алгоритма и фазы обработки, а не дублировать каждую инструкцию
5. Краткий комментарий справа у строки нужен только там, где код реально неочевиден: плотная арифметика, оптимизация, хитрый регистровый контракт, магическая константа, нетривиальный переход.

## Code Style

- Именование: `lower_snake_case` для меток, процедур, констант, полей структур, таблиц, буферов и модулей.
- Метки процедур и точек перехода писать отдельной строкой, без двоеточия, с нулевого столбца.
- Внутренние метки именовать с префиксом по функции или по смыслу (`update_room_next`, `jump_process_left4`), чтобы не плодить безымянные `loop1`.
- Инструкции, директивы и операнды писать в нижнем регистре: `ld`, `call`, `equ`, `db`, `dw`, `ds`, `include`, `incbin`.
- Для новых строк кода в asm держать `28` пробелов от начала строки до мнемоники.
- После мнемоники использовать один `tab` как разделитель до операнда или аргумента.
- Метки по-прежнему писать с нулевого столбца, без отступа.
- `module ...` / `endmodule` использовать как основной контейнер файла или подмодуля.
- `struct ...` / `ends` использовать для описания packed-структур; порядок полей считать частью контракта.
- Крупные подсистемы раскладывать на связанные `include`-файлы рядом с корневым модулем: как в `elevators_*.a80`, `rooms_*.a80`, `logic/*.a80`.
- Таблицы и буферы именовать по назначению: `*_table`, `*_buf`, `*_end`, `*_end_marker`, `*_amount`, `*_lim`.
- Функцию не писать сплошной простынёй. Разбивать на смысловые блоки по 2-10 команд: чтение входов, проверки, основное действие, сохранение результата, выход.
- Недвусмысленное правило по пустым строкам: между соседними логическими блоками внутри процедуры оставлять ровно одну пустую строку. То есть между последней непустой строкой блока A и первой непустой строкой блока B должно быть два символа перевода строки подряд (`\n\n`).
- Если начинается новая фаза алгоритма, нужен новый блок. Если несколько команд делают одно короткое действие, держать их в одном блоке.
- Комментарии писать коротко и по делу, обычно перед блоком. Правые комментарии оставлять только в сложных местах или длинных блоках.
- Для структур, инициализаторов и таблиц допустимы короткие комментарии схемы данных, но не шум на каждой строке без необходимости.
- Для инициализаторов структур хорошая практика: ставить справа краткие комментарии с именами полей (`; room_x`, `; delay`, `; frames_ptr`), чтобы порядок параметров читался без сверки со `struct`.
- Новые комментарии в asm писать по-русски, чтобы не смешивать стили внутри файла.

## Comment Language

- New block comments in asm should be brief and in English.
- Prefer one short comment per semantic block, not per instruction.
- Right-side comments should be used only for non-obvious code.

## Platform

- Целевая платформа проекта: `ZX Spectrum 128`.
- Сборка идёт через `sjasmplus`: `compile.bat` запускает `_sjasmplus\\sjasmplus.exe main.a80`.
- Точка входа программы находится в `main.a80`, результат сборки сейчас сохраняется как `qsave1.sna`.
- Все изменения должны оставаться совместимыми с текущими модулями экрана, комнат и отрисовки (`screen_utils`, `draw`, `rooms`).
- Секция `Platform` зарезервирована и дальше может дополняться адресацией памяти, устройством экрана, атрибутами и прочими аппаратными правилами.

## Domain

- Это игра с героем, который бегает по экранным комнатам и переходит между ними.
- В проекте уже есть лифты, предметы, анимации комнаты, телепорты, водяные капли и отдельная логика комнат.
- Типовой паттерн домена: есть общая таблица сущностей, а при инициализации комнаты в буфер собираются только указатели на сущности текущей комнаты.
- Новые комнатные сущности и механики по возможности встраивать в этот паттерн, а не обходить его одноразовым хардкодом.
- При изменении логики помнить, что координаты и поведение героя связаны с комнатой, экраном, состояниями движения и буферами текущей комнаты.
